# はじめに

## 目的

本書は、社内向け在庫管理システムが提供する REST API(以下、「本 API」という)の仕様を定めることを目的とする。本書を参照することで、クライアントアプリケーションの開発者は本 API を正しく利用し、また API 提供側の開発者は仕様変更の影響範囲を正確に把握できることを目指す。

在庫管理は日々の入出庫、棚卸、発注といった業務と密接に関わるため、仕様の一字一句が現場のオペレーションに直結する。そのため本書では、曖昧さを避けるために可能な限り具体的なリクエスト例・レスポンス例を添えて説明する。

## 適用範囲

本書が対象とする範囲は、在庫管理システムのバックエンドが公開する REST API の外部インタフェースに限定される。データベースの物理設計、内部バッチ処理の実装詳細、社内認証基盤(SSO)の内部仕様については、それぞれ別紙の設計書を参照すること。

## 用語定義

本書で用いる主要な用語を次の表に示す。

| 用語 | 説明 |
|---|---|
| SKU | Stock Keeping Unit の略。在庫管理上の最小識別単位。 |
| ロケーション | 倉庫内における在庫の保管場所を一意に示す識別子。 |
| 引当 | 受注に対して出荷予定の在庫を確保する処理。 |
| 棚卸差異 | 帳簿上の在庫数と実地棚卸で確認した実在庫数との差分。 |

: 用語定義一覧

# システム概要

在庫管理システムは、複数の倉庫拠点にまたがる在庫情報を一元的に管理し、受発注システムや会計システムと連携するための基盤である。本 API はそのうち「在庫照会」「入出庫登録」「棚卸」の 3 つの業務領域を提供する。

システムは大きく分けて次のコンポーネントから構成される。

| コンポーネント | 役割 |
|---|---|
| API ゲートウェイ | 認証・認可、レートリミット、リクエストのルーティングを担当する。 |
| 在庫サービス | 在庫数量・ロケーション情報の管理と更新を担当する中核サービス。 |
| イベントバス | 在庫変動イベントを他システム(受発注・会計)へ非同期に配信する。 |
| 監査ログストア | すべての在庫変動操作を追跡可能な形式で永続化する。 |

: システム構成コンポーネント

外部システムとの連携は、原則として本 API 経由の同期呼び出し、または在庫変動イベントの非同期購読のいずれかによって行う。データベースへの直接アクセスは、いかなる場合も許可されない。

# 機能要件

本章では、本 API が満たすべき機能要件を要件 ID とともに列挙する。

| 要件ID | 要件概要 | 優先度 |
|---|---|---|
| FR-001 | SKU 単位・ロケーション単位で在庫数量を照会できること | 必須 |
| FR-002 | 入庫・出庫のイベントを記録し、在庫数量へ即時反映すること | 必須 |
| FR-003 | 在庫引当および引当解除を行えること | 必須 |
| FR-004 | 棚卸差異を記録し、差異理由コードを付与できること | 推奨 |
| FR-005 | 在庫変動履歴を期間指定で取得できること | 推奨 |

: 機能要件一覧

在庫引当の処理フローは、次の手順に従うものとする。

1. 受注情報を受け取り、対象 SKU の引当可能数量を確認する。
2. 引当可能数量が要求数量以上であれば、引当済み数量を加算し引当 ID を発行する。
3. 引当可能数量が不足する場合は、エラーコード `INSUFFICIENT_STOCK` を返却する。
4. 出荷確定後、引当済み数量を実在庫の減算に振り替える。
5. 出荷キャンセル時は、引当を解除し引当可能数量へ戻す。

> **注意:** 引当処理は必ず単一トランザクション内で完結させること。引当可能数量の確認から引当済み数量の更新までの間に他のリクエストが割り込むと、二重引当による過剰出荷が発生するおそれがある。悲観ロックまたは楽観ロック(バージョン番号による排他制御)のいずれかを必ず用いること。

なお、引当可能数量の再計算ロジックは非常に重要度が高く、変更を行う際は必ずレビューを二名以上で実施すること[^review-policy]。

[^review-policy]: 過去に引当ロジックの不具合により在庫過剰引当が発生した事例があるため、社内ルールとしてダブルチェックを義務付けている。

# API仕様

本章では、本 API が提供する主要なエンドポイントについて説明する。ベース URL は `https://api.example.internal/inventory/v1` とする。

## エンドポイント一覧

| メソッド | パス | 概要 |
|---|---|---|
| GET | `/items/{sku}` | 指定した SKU の在庫情報を取得する |
| GET | `/items` | 在庫一覧を検索条件付きで取得する |
| POST | `/items/{sku}/reservations` | 在庫引当を作成する |
| DELETE | `/items/{sku}/reservations/{id}` | 在庫引当を解除する |
| POST | `/stocktakes` | 棚卸結果を登録する |

: API エンドポイント一覧

## 在庫情報の取得

`GET /items/{sku}` は、指定した SKU の在庫情報をロケーション単位で返却する。レスポンスの例を次に示す。

```json
{
  "sku": "SKU-00012345",
  "name": "段ボール箱(A4サイズ)",
  "locations": [
    { "location": "WH1-A-001", "available": 120, "reserved": 15 },
    { "location": "WH2-B-014", "available": 40, "reserved": 0 }
  ],
  "updatedAt": "2026-07-14T09:30:00+09:00"
}
```

curl での呼び出し例は次のとおりである。

```bash
curl -sS \
  -H "Authorization: Bearer ${ACCESS_TOKEN}" \
  "https://api.example.internal/inventory/v1/items/SKU-00012345"
```

レスポンスのステータスコードが `200` 以外の場合、クライアントは `error.code` フィールド(例: `SKU_NOT_FOUND`)を参照してエラー種別を判定すること。SKU の指定には `sku` パスパラメータを用い、インラインコードで表すと `sku` は常に `SKU-` で始まる 13 桁の識別子である。

## 在庫引当の作成

`POST /items/{sku}/reservations` は、指定した SKU に対して在庫引当を作成する。リクエストボディの主なフィールドを次に示す。

| フィールド | 型 | 必須 | 説明 |
|---|---|---|---|
| `quantity` | integer | 必須 | 引当数量。1 以上の整数。 |
| `orderId` | string | 必須 | 引当元となる受注 ID。 |
| `location` | string | 任意 | 引当対象のロケーション。省略時は自動選択。 |

: 引当作成リクエストのフィールド

# 非機能要件

本 API は、次に示す非機能要件を満たすものとする。

- **可用性**: 月間稼働率 99.9% 以上を維持すること。計画メンテナンスは事前告知のうえ深夜帯に実施する。
- **性能**: `GET /items/{sku}` の応答時間は、95 パーセンタイルで 200 ミリ秒以内とすること。
- **拡張性**: ロケーション数の増加に対して線形に近いスケールアウトが可能な設計とすること。
- **監査性**: すべての在庫変動操作は、操作者・操作日時・変更前後の値を監査ログへ記録すること。
- **セキュリティ**: すべての通信は TLS 1.2 以上で暗号化し、アクセストークンの有効期限は最長 1 時間とすること。

これらの非機能要件は四半期ごとにレビューを行い、実測値との乖離が大きい項目については改善計画を別途策定するものとする。特に性能要件については、繁忙期(月末月初)のトラフィック増加を考慮した負荷試験を年に一度実施し、結果を本書の改訂履歴に反映すること。

# 付録A: エスケープハッチの例

本書はほとんどの内容を素の Markdown で記述しているが、セル結合を伴う表など Markdown 標準の記法では表現できない場合に限り、Pandoc の生 Typst 記法(`` ```{=typst} `` フェンス)を用いてよい。乱用は禁物であり、あくまで最終手段として使うこと。

次の表は、ロケーション別の棚卸結果をセル結合で表現した例である。

```{=typst}
#figure(
  table(
    columns: 4,
    align: (center + horizon, center + horizon, center + horizon, left + horizon),
    table.header([倉庫], [ロケーション], [差異数], [備考]),
    table.hline(),
    table.cell(rowspan: 2)[第1倉庫],
    [WH1-A-001], [-2], [棚卸時に破損品を発見],
    [WH1-A-002], [0], [差異なし],
    table.cell(rowspan: 2)[第2倉庫],
    [WH2-B-014], [+5], [入庫未登録分を確認],
    [WH2-B-015], [-1], [所在不明],
  ),
  caption: [棚卸差異一覧(ロケーション別)],
  kind: table,
)
```

上記のように、生 Typst ブロック内では `table.cell(rowspan: ...)` によるセル結合のみを書けばよく、罫線やヘッダの配色といった見た目は `spec.typ` の `show table: ...` が自動的に適用する。これは `template/template.typ` が `#import "/template/spec.typ": *` によってテーマの定義一式を取り込んでいるためであり、色定数(`accent-color` など)やヘルパー関数も必要であればそのまま参照できる。
