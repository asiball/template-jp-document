本書は、`template-jp-document` を使って Markdown から仕様書 PDF を作るための、非技術者向けの最短手順です。Markdown に不慣れな PM・品証の方を対象としています。執筆ルールやビルドの仕組みの詳細は `README.md` を参照してください。

## 見本をコピーして書き始める

一から書き始めるのではなく、見本をコピーして、中身を自分の仕様書の内容に置き換えるのが最も早い方法です。見本は 2 種類あり、文書の分量に応じて選びます。

### 単一ファイルで始める（`wareki-api-spec.md` 参考）

短め〜中くらいの分量であれば、1 つの Markdown ファイルに全部書く方式で十分です。見本フォルダ `examples/` の `wareki-api-spec.md` を、自分の原稿置き場である `docs/` へコピーして使います。

```sh
cp examples/wareki-api-spec.md docs/my-spec.md
```

ファイル名（`my-spec` の部分）は半角の英数字とハイフンで自由に付けられます（日本語ファイル名は環境によって文字化けやビルドの不具合の原因になりうるため避けてください）。見出し・表・コードブロックといった「型」はすでに見本の中にあるので、内容だけを書き換えれば体裁は自動的に整います。

### 大きな文書は章分割（`sample-spec` 参考）

章の数が多くなり 1 ファイルでは編集しづらくなってきたら、章ごとにファイルを分ける方式に切り替えられます。`examples/sample-spec/` フォルダを `docs/` へコピーして使います。

```sh
cp -r examples/sample-spec docs/my-spec
```

このフォルダの中身は次のようになっています。

```
docs/my-spec/
├── 00-meta.md              フロントマター(タイトルなどの情報)専用ファイル
├── 01-introduction.md      1 章目
├── 02-overview.md          2 章目
└── ...                     以下、章ごとにファイルが分かれている
```

- **`00-meta.md`** には、次節で説明する「表紙・ヘッダに表示する情報」だけを書きます。本文の章は書きません。
- 章ファイルの**ファイル名の並び順がそのまま章の並び順**になります。数字の後ろは好きな名前を付けられます（前述のとおり、ファイル名・フォルダ名には半角の英数字とハイフンを使い、日本語は避けてください）。
- 章ファイルには `00-meta.md` のような情報（タイトルなど）を書かないでください。書いてしまうと、後ろの章ファイルの情報が前の章ファイルの情報を上書きしてしまう事故につながります。

どちらの方式でビルドしても、できあがる PDF の見た目は同じです。迷ったら、まず 1 ファイル方式で書き始め、後から章分割に切り替えても構いません。

## 冒頭の YAML メタデータを書く

ファイル先頭の `---` で囲まれた部分に、表紙・ヘッダに表示する情報を書きます（章分割の場合は `00-meta.md` の先頭に書きます。章分割した場合、章ファイル（`01-introduction.md` など）にはこの `---` ブロックを書かないでください）。

| 変数 | 必須 | 内容 |
|---|---|---|
| `title` | 必須 | 表紙に大きく表示するタイトル |
| `subtitle` | 任意 | サブタイトル |
| `docnumber` | 任意 | 文書番号 |
| `version` | 任意 | 版数 |
| `date` | 任意 | 発行日 |
| `author` | 任意 | 作成者 |
| `organization` | 任意 | 発行組織名 |
| `revisions` | 任意 | 改訂履歴（バージョン・日付・作成者・変更内容の一覧） |

各変数の詳しい説明は `README.md` の「メタデータ(YAML フロントマター)一覧」を参照してください。

**補足**: `revisions`（改訂履歴）が版を重ねて長くなってきたら、別ファイルに切り出せます（1 ファイル方式なら `docs/my-spec.revisions.md`、章分割方式なら `docs/my-spec/revisions.md`）。中身は普段の Markdown の表と同じ書き方で、改訂を重ねるたびに末尾へ 1 行足していくだけです（ファイルを置けば自動的に読み込まれます）。

```markdown
| 版数 | 日付 | 作成者 | 改訂内容 |
|---|---|---|---|
| 1.0 | 2026-05-01 | 山田太郎 | 初版作成 |
| 1.1 | 2026-06-10 | 鈴木花子 | 在庫一覧の章を追加 |
```

このとき、仕様書本体のフロントマターからは `revisions:` の項目を削除してください（両方に書くと別ファイル側が無視されます）。詳しくは `README.md` の「改訂履歴の別ファイル化」を参照してください。

## 見出しを書く（番号は自動）

`#` の数がそのまま見出しの階層になります。

```markdown
# 章のタイトル
## 節のタイトル
### 項のタイトル
```

**番号は自分で書かないでください**。`# 1. はじめに` のように自分で番号を振ると、PDF 化したときに「1 1. はじめに」のように番号が二重になってしまいます。番号は PDF 化の際に自動で付きます。

## 表・コード・図・脚注の最小例

表は Markdown 標準の書き方でそのまま書けます。

```markdown
| 項目 | 内容 |
|---|---|
| 対象読者 | 非技術者の PM・品証 |
```

コードブロックは、開始と終了をバッククォート3つで囲み、直後に言語名を書きます。

```json
{ "example": true }
```

図は `/assets/images/` に画像ファイル（PNG/JPG/SVG）を置き、リポジトリルートからの絶対パスで参照します。

```markdown
![構成図の説明文](/assets/images/example.png)
```

シーケンス図や状態遷移図などは、PlantUML というテキスト記法のソースファイル（`.puml`）を `/assets/diagrams/` に置いて同じように参照すると、ビルド時に自動で図に変換されます（画像ファイルを自分で作る必要はありません。詳細は [README](README.md) の「図の挿入」節を参照）。

```markdown
![在庫引当作成の処理シーケンス](/assets/diagrams/reservation-sequence.puml)
```

脚注は本文中に番号を置き、同じ番号の内容を文末にまとめて書きます。

```markdown
本文中の注記です[^1]。

[^1]: 脚注の内容。
```

## ビルドして PDF を確認する

自分のパソコンに pandoc / typst が入っていれば、次のコマンドで PDF を作れます。

```sh
make pdf SRC=docs/my-spec.md    # 1 ファイル方式の場合
make pdf SRC=docs/my-spec       # 章分割方式の場合(.md を付けない)
```

環境構築が難しい場合は、開発メンバーやビルド担当者に「`docs/my-spec.md`（または `docs/my-spec/`）をビルドしてほしい」と依頼してください。

## やってはいけないこと

- **フォント・色・余白などのスタイル指定を書かない**。太字・表・コードブロックといった「構造」だけを書けば、見た目は自動的に整います。見た目そのものを変えたい場合は開発メンバーに相談してください。
- **見出しに自分で番号を振らない**（前述のとおり、自動採番と二重になります）。
